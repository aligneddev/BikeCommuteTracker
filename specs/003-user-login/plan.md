# Implementation Plan: User Login with PIN

**Branch**: `003-user-login` | **Date**: 2026-03-17 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-user-login/spec.md`

## Summary

Deliver the user login vertical slice: split the existing combined Create User / Identify page into two independent pages, rename Identify to Login, introduce client-side auth state (React Context + sessionStorage), protect the miles shell route via a `ProtectedRoute` wrapper, and create a minimal miles placeholder page as the post-login destination. No new backend endpoints are added; the existing `/api/users/identify` is reused for the login action. `react-router-dom` v7 is added to enable multi-page navigation.

## Technical Context

**Language/Version**: TypeScript 5.x (React 19 + Vite); .NET 10 C# / F# backend (unchanged)  
**Primary Dependencies**: `react-router-dom` v7 (new); React 19, Vite, ASP.NET Core Minimal API (existing)  
**Storage**: `sessionStorage` (client-side auth session only); SQLite via EF Core (existing, unchanged)  
**Testing**: ESLint + TypeScript compiler for static checks; Playwright MCP for E2E user journeys; xUnit for backend (unchanged)  
**Target Platform**: Desktop and mobile browsers; local Aspire-orchestrated stack  
**Project Type**: Aspire-orchestrated web application (React SPA frontend + .NET Minimal API + SQLite)  
**Performance Goals**: Page navigation <100ms (client-side router); API login response <500ms p95  
**Constraints**: Local-only scope; no server-side session management; no persistent login across browser sessions; frontend-only rename of "Identify" to "Login"  
**Scale/Scope**: Single local deployment; three-page React SPA (Login, Create User, Miles Shell)

## Phase 0 Research Outcomes

Research decisions are documented in [research.md](research.md) and resolve the following:

- **Routing**: `react-router-dom` v7 selected for multi-page navigation (Login / Create User / Miles Shell).
- **Auth state**: React Context + `sessionStorage` for single-session login state; no server-side sessions.
- **Protected route**: Single `<ProtectedRoute />` layout component using React Router `<Outlet />`.
- **API reuse**: `/api/users/identify` reused for login; `loginUser` alias added to `users-api.ts`; no backend changes.
- **Miles shell**: Minimal placeholder page showing authenticated user's name and a logout button.
